# 需求文档：workspace-frpc-isolation

## 功能概述

将桌面端的单一聚合 `frpc` 改为每个工作区一个独立 `frpc`，工作区内 MCP 与 Actions 共享该实例。系统持久化工作区的本地端口和 FRP 配置，并在启动本地服务或隧道前检查端口、代理名和子域名冲突；冲突时阻止启动并指出占用资源的工作区与服务。

## 历史经验与坑

- **可复用经验**：继续使用 `(workspace_id, service)` 作为线路归属键，配置替换采用“暂存旧状态、启动新状态、失败恢复”的事务式流程。
- **必须规避的坑**：不能按 `frpc` 可执行文件路径批量结束进程；不能让一个工作区的线路变更重启其他工作区；不能只依赖操作系统端口占用结果而忽略工作区配置冲突；`frpc` 退出后 FRPS 释放旧代理存在延迟，回滚和重启必须有界重试。

## 术语定义

- **工作区 frpc**：仅承载一个工作区 MCP/Actions FRP 线路的 `frpc` 进程。
- **资源声明**：工作区配置对本地端口、FRP proxy name 和 HTTP subdomain 的占用记录。
- **启动前校验**：在创建本地监听器或 `frpc` 进程前执行的持久化配置冲突检查。

---

## 范围边界

**In Scope（本次要做）**

- 每个工作区独立管理 `frpc` Child、PID、配置文件和操作锁。
- 工作区内 MCP 与 Actions 使用同一 `frpc`，且必须连接同一 FRPS 地址、端口和 Token。
- 本地 MCP/Actions 端口在全部工作区和服务之间唯一。
- 启动本地服务和显式启动/测试隧道前执行端口、proxy name、subdomain 冲突校验。
- 冲突错误包含冲突值、占用工作区名称和服务名称，并由现有页面 Toast 直接展示。
- 停止服务时只变更当前工作区 `frpc`；工作区还有另一条线路时重建当前工作区实例，无线路时停止该实例。
- Windows、macOS、Linux 使用同一工作区归属规则。

**Out of Scope（本次不做）**

- 不修改 FRPS 服务端配置或认证协议。
- 不支持 TCP/UDP `remotePort` 代理；当前产品仅生成 HTTP subdomain 代理。
- 不允许不同工作区有意复用同一本地端口进行互斥启动。
- 不管理用户在桌面应用之外手工启动的 `frpc`。

## 需求列表

### FR-1：每工作区独立 frpc 生命周期

**优先级：** Must

**用户故事：** 作为多工作区用户，我想让每个工作区拥有独立 `frpc`，以便一个工作区启停或失败时不影响其他工作区。

#### 验收标准（EARS）

1. WHEN 首次启动工作区 FRP 线路 THEN 系统 SHALL 只创建该工作区的 `frpc` 进程和配置文件。
2. WHILE 其他工作区 `frpc` 正在运行 THE 系统 SHALL 不停止、覆盖配置或改变其状态。
3. IF 当前工作区没有剩余 FRP 线路 THEN 系统 SHALL 只停止当前工作区 `frpc`。

### FR-2：工作区内双服务共享

**优先级：** Must

**用户故事：** 作为工作区用户，我想让 MCP 与 Actions 共用一个工作区 `frpc`，以便在隔离性和进程数量之间保持平衡。

#### 验收标准（EARS）

1. WHEN 当前工作区 MCP 与 Actions 均启用 FRP THEN 系统 SHALL 在同一个工作区配置中生成两个唯一代理。
2. WHEN 停止其中一个服务 THEN 系统 SHALL 重建当前工作区 `frpc` 并保留另一个代理。
3. IF 两个服务连接不同 FRPS 地址、端口或 Token THEN 系统 SHALL 阻止第二条线路启动并说明原因。

### FR-3：端口记录与启动前冲突阻断

**优先级：** Must

**用户故事：** 作为用户，我想在点击启动时立即知道端口被哪个工作区占用，以便先修正配置而不是等待启动失败。

#### 验收标准（EARS）

1. WHEN 保存工作区配置 THEN 系统 SHALL 持久化 MCP 与 Actions 的本地端口。
2. WHEN 点击启动 MCP、Actions 或对应隧道 THEN 系统 SHALL 在创建进程前检查全部工作区的端口声明。
3. IF 同一端口被另一个工作区或当前工作区另一服务声明 THEN 系统 SHALL 阻止启动，并返回“端口、工作区、服务”三项信息。
4. IF 端口未被配置声明但已被外部进程监听 THEN 系统 SHALL 保留现有 PID 占用错误。

### FR-4：FRP 代理资源唯一性

**优先级：** Must

**用户故事：** 作为用户，我想在启动前发现代理名或子域名冲突，以便避免 FRPS 返回 `proxy already exists`。

#### 验收标准（EARS）

1. WHEN 启动或测试 FRP 线路 THEN 系统 SHALL 校验同一 FRPS 上的 proxy name 和 HTTP subdomain 唯一。
2. IF 发生冲突 THEN 系统 SHALL 阻止启动并返回冲突工作区、服务和冲突值。
3. WHEN 生成 proxy name THEN 系统 SHALL 使用稳定工作区 ID 参与命名，避免同名工作区产生相同代理名。

### FR-5：进程归属与失败恢复

**优先级：** Must

**用户故事：** 作为用户，我想让应用只停止自己记录的工作区 `frpc`，以便其他线路和外部进程保持安全。

#### 验收标准（EARS）

1. WHEN 启动工作区 `frpc` THEN 系统 SHALL 按工作区记录 PID、配置路径和日志归属。
2. WHEN 停止或恢复工作区线路 THEN 系统 SHALL 按记录 PID 且核对镜像路径后终止进程，不得按镜像路径批量终止全部 `frpc`。
3. IF FRPS 暂未释放旧代理并返回 `proxy already exists` THEN 系统 SHALL 在当前工作区范围内有界退避重试，且不得影响其他工作区。
4. IF 新配置最终启动失败 THEN 系统 SHALL 恢复当前工作区旧 route，并返回包含原始错误和回滚结果的明确错误。

## 非功能需求

- **NFR-1（性能）**：资源冲突检查在 500 个工作区内应于 50ms 内完成，不访问网络。
- **NFR-2（安全）**：PID 终止前必须确认进程仍存活且镜像路径属于应用解析到的 `frpc`。
- **NFR-3（兼容性）**：Windows x64、macOS x64/ARM64、Linux x64/ARM64 保持相同行为。
- **NFR-4（可靠性）**：所有启动入口均执行后端二次校验，前端提示不能作为唯一门禁。

## 依赖关系

- `WorkspaceProfile` 已持久化 MCP/Actions 本地端口及 FRP 配置。
- `RuntimeSupervisor` 管理本地服务生命周期。
- `TunnelSupervisor` 管理 route、session 与 `frpc` 生命周期。
- 平台层提供 PID 存活、镜像路径查询和进程树终止能力。
